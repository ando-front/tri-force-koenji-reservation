# Tri-force Koenji施設予約システム 運用手順書

## 1. 目的

本書は Firebase Hosting + Cloud Functions + Firestore 構成の現行システムについて、リリース再実行、本番確認、費用見積もりの基準を整理するための運用手順書である。

## 2. 前提条件

### 2.1 本番運用の前提

- Firebase プロジェクトは Blaze プランであること
- GitHub Secrets が登録済みであること
- `FIREBASE_SERVICE_ACCOUNT` が Hosting デプロイ可能であること
- `FIREBASE_CI_TOKEN` が Functions デプロイ可能であること
- `RESEND_API_KEY` と `MAIL_FROM` が有効であること
- 管理画面で使用する Google アカウントが Firebase Authentication 側で管理者として許可されていること

### 2.2 Blaze 切り替え後に確認する項目

1. Firebase コンソールで対象プロジェクトの課金プランが Blaze に変わっている
2. Google Cloud 請求先アカウントが対象プロジェクトに紐付いている
3. `cloudbuild.googleapis.com` と `artifactregistry.googleapis.com` が有効化済み、または有効化可能な状態になっている
4. 反映待ち時間として 5 分から 10 分程度は空ける

## 3. GitHub Actions 再実行手順

### 3.1 対象ワークフロー

- 本番 CI: `.github/workflows/ci.yml`
- 本番 Deploy: `.github/workflows/deploy.yml`
- staging Deploy: `.github/workflows/staging.yml`

### 3.2 今回の再実行方針

現時点の最新本番対象コミットは `d63c01a` であり、CI は成功済み、Deploy のみが Blaze 未切り替えで失敗した。

そのため Blaze 切り替え後は次の順で再実行する。

1. 既存の `Deploy` 失敗実行を再実行する
2. 成功後に `Staging Deploy` を手動実行する
3. staging で動作確認後、本番 URL で確認する

### 3.3 GitHub 画面からの再実行

#### A. 本番 Deploy の再実行

1. GitHub リポジトリの `Actions` を開く
2. `Deploy` ワークフローを開く
3. 失敗している最新実行を開く
4. `Re-run all jobs` を押す
5. `Build & Deploy to Firebase` ジョブで次を確認する
   - `Build Functions` が成功
   - `Build Frontend` が成功
   - `Deploy to Firebase` が成功
   - `Deploy Functions` が成功

#### B. staging Deploy の手動実行

1. `Actions` から `Staging Deploy` を開く
2. `Run workflow` を押す
3. Branch は `main` を選ぶ
4. 実行後、Hosting staging URL と Functions デプロイ成功を確認する

### 3.4 GitHub CLI からの再実行

同じ内容は `gh` でも実行できる。

```bash
# 失敗した本番 Deploy を再実行
gh run rerun 23373692158

# 最新状態を監視
gh run watch 23373692158

# staging ワークフローを main で手動起動
gh workflow run staging.yml --ref main
```

### 3.5 再実行時の判定基準

- `CI` が green
- `Deploy` が green
- Hosting の live channel に新ビルドが反映
- Functions デプロイで Blaze/Cloud Build 関連エラーが消えている

### 3.6 再実行後にまだ失敗する場合の確認順

1. Blaze 切り替えが対象 Firebase プロジェクトに対して行われているか
2. `FIREBASE_CI_TOKEN` の発行元アカウントに対象プロジェクトの権限があるか
3. `FIREBASE_SERVICE_ACCOUNT` が古い JSON のままではないか
4. GitHub Secrets 更新後に再実行しているか

### 4.3 予約作成確認

#### 手順

1. 公開トップページを開く
2. 施設を選択する
3. 空き時間を選択する
4. 次の項目を入力して送信する
   - 氏名
   - メールアドレス
   - 参加人数
   - 利用目的
   - 備考
5. 完了画面に遷移することを確認する

#### 合格条件

- エラーなく送信できる
- 完了画面に予約番号が表示される
- 予約番号先頭 8 文字が表示される
- 予約内容が入力値と一致する

### 4.4 管理画面確認

#### 手順

1. 管理画面ログインページを開く
2. 管理者 Google アカウントでログインする
3. 予約一覧で新規予約が表示されることを確認する
4. 予約詳細を開き、氏名、メール、日時、人数、目的、備考を確認する
5. 可能なら `pending` から `confirmed` へ更新する

#### 合格条件

- ログインできる
- 新規予約が一覧に存在する
- 詳細画面の項目が完了画面と一致する
- ステータス更新が成功し、一覧にも反映される

### 4.5 CSV 出力確認

#### 手順

1. 管理画面一覧で必要に応じて日付や施設で絞り込む
2. `CSVエクスポート` を押す
3. ダウンロードされた CSV を開く
4. 予約番号、会員名、メールアドレス、施設名、日付、開始時刻、終了時刻、参加人数、利用目的、備考、ステータス、登録日時を確認する

#### 合格条件

- CSV がダウンロードされる
- 文字化けしない
- 対象予約が含まれる
- フィルター条件が CSV に反映される

### 4.6 メール送信確認

#### 手順

1. 予約作成で使用したメールアドレスの受信箱を確認する
2. 件名に予約番号先頭 8 文字が含まれていることを確認する
3. 本文に施設名、利用日、時間、参加人数、利用目的が含まれることを確認する
4. 届かない場合は Resend ダッシュボードまたは Functions ログを確認する

#### 注意点

- メール送信は非同期であり、予約 API はメール送信完了を待たず成功を返す
- そのため画面成功とメール成功は分けて確認する

#### 合格条件

- 数分以内にメールが届く
- 件名と本文に予約内容が含まれる
- 必要なら `ADMIN_MAIL_BCC` 宛てにも通知が飛ぶ

## 7. トラブルシューティング（参考）

### 施設が表示されない場合

**確認順:**

1. Browser DevTools の Console タブで js error がないか
2. Network タブで `/facilities` API 呼び出しが 200 OK か、error か
3. Firestore コンソール → `facilities` コレクション存在確認
4. Firebase Functions ログで API エラーを確認

**よくあるケース:**
- Firestore に facilities データなし → 空配列 200 OK（仕様）
- admin.firestore 权限不足 → API 500 error
- Functions 環境変数 VITE_FUNCTIONS_BASE_URL 未設定 → fallback に自動切り替え

- Functions は `asia-northeast1`、`256MiB`、`maxInstances: 10`
- 予約 1 件につき API 呼び出しは 2 回から 4 回程度
  - 施設一覧取得
  - 空き状況取得
  - 予約作成
  - 管理確認や CSV は別途
- 確認メールは予約 1 件につき 1 通
- Resend は月 3,000 通までは無料枠想定

### 5.2 シナリオ別の概算

| シナリオ | 月間予約数 | 月間ページ表示 | 月間メール数 | 概算月額 |
|---|---:|---:|---:|---:|
| 小規模運用 | 300件 | 3,000PV | 300通 | ¥0〜¥300 |
| 標準運用 | 1,000件 | 10,000PV | 1,000通 | ¥0〜¥800 |
| 繁忙運用 | 3,000件 | 30,000PV | 3,000通 | ¥500〜¥2,000 |

### 5.3 内訳の見方

- Hosting
  - 小規模なら無料枠またはごく少額で収まる可能性が高い
- Firestore
  - 予約一覧や空き状況の読み取りが増えると増額する
  - ただし本システム規模では大きな支配要因になりにくい
- Functions
  - 実行回数よりも大量アクセス時のリクエスト増加と外部通信で増えやすい
- Resend
  - 3,000 通を超えなければ無料枠内想定

### 5.4 重要な注意

- Blaze への切り替えは課金の許可であり、即高額課金ではない
- 今回の構成ではランニングコストよりも `Functions をデプロイ可能にするため Blaze が必要` という意味合いが大きい
- 正確な請求額は Firebase / Google Cloud / Resend の実測値で確定する

### 5.5 運用開始時に必ず設定するもの

1. Google Cloud 予算アラート
2. Firebase 使用量アラート
3. Resend 使用量確認
4. 月次で Hosting 転送量、Firestore 読み取り回数、Functions 実行回数を確認
