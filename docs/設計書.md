**バージョン**: 2.0（再構築版）

**対象構成**: Firebase Hosting + Cloud Functions + Firestore

## 1. システムアーキテクチャ

### 1.1 構成方針

現行GAS + Googleスプレッドシート構成の課題（複数アカウント問題・実行時間制限・テスト不可・属人化）を解消するため、以下の原則で再設計する。

- **サーバーレス**: 常時稼働サーバーを持たない（コスト・運用負担ゼロ）
- **無償枠内**: Firebase無料枠（Spark plan）で月間通常運用を完結させる
- **型安全**: TypeScriptを全層で統一し、コンパイル時にバグを検出
- **テスト可能**: ビジネスロジックを関数として独立させ、JestでCI実行
- **Git中心**: インフラもコードもGitHub管理、GitHub Actionsで自動デプロイ

### 1.2 技術スタック

| 層 | 技術 | 理由 |
|---|---|---|
| フロントエンド | Vite + React 18 + TypeScript | 軽量・モダン・標準的 |
| スタイル | Tailwind CSS v3 | ユーティリティファーストで保守しやすい |
| ホスティング | Firebase Hosting | 無償・CDN・SSL自動・カスタムドメイン対応 |
| API | Firebase Cloud Functions v2 (Node.js 20, TypeScript) | トリガー不要の単純HTTPに最適 |
| データベース | Cloud Firestore | スキーマレス・リアルタイム・無償枠大 |
| 認証 | Firebase Authentication（Googleログイン） | 管理画面のみ・設定ゼロ |
| メール通知 | Resend（無料枠: 3,000通/月） | API経由・SPF/DKIM対応 |
| CI/CD | GitHub Actions | mainブランチpushで自動テスト＋デプロイ |
| 監視 | Firebase Crashlytics + Cloud Logging | 無償枠内 |

```typescript
// /facilities/{facilityId}
interface Facility {
  facilityId: string;        // ドキュメントIDと同一
  name: string;              // 例: "フリーマット", "フィットネス"
  capacity: number;          // 同時予約上限人数
  openHour: number;          // 営業開始時刻（整数時）例: 7
  closeHour: number;         // 営業終了時刻（整数時）例: 21
  slotDurationMinutes: number; // 予約単位（分）例: 60
  closedWeekdays: number[];  // 定休曜日 0=日〜6=土
  maintenanceDates: string[]; // メンテナンス日 "YYYY-MM-DD"
  weekdayHours?: WeekdayHours[]; // 曜日ごとの営業時間（省略時は共通設定を使用）

  updatedAt: Timestamp;

  openHour: number;
  closeHour: number;

}
```

### 2.7 Firestoreセキュリティルール

```
// firestore.rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // 施設マスタ: 読み取り自由、書き込みは管理者のみ
    match /facilities/{facilityId} {
      allow read: if true;
      allow write: if isAdmin();
    }

    // 予約: 書き込みは Functions 経由（クライアントからの直接書き込み不可）
    // 読み取りも Functions 経由を基本とする
    match /reservations/{reservationId} {
      allow read:  if isAdmin();
      allow write: if false;  // Functions の Admin SDK のみ許可
    }

    // 管理者: 自分のドキュメントのみ読み取り可
    match /admins/{uid} {
      allow read:  if request.auth != null && request.auth.uid == uid;
      allow write: if false;
    }

    // 監査ログ: 読み書き不可（Admin SDK のみ）
    match /auditLogs/{logId} {
      allow read:  if false;
      allow write: if false;
    }

    function isAdmin() {
      return request.auth != null
        && exists(/databases/$(database)/documents/admins/$(request.auth.uid));
    }
  }
}
```

### 3.1 エンドポイント一覧

ベースURL: `https://{region}-{projectId}.cloudfunctions.net/api`

| `GET` | `/reservations/admin/:id` | 管理者 | 予約詳細 |
| `PATCH` | `/reservations/admin/:id/status` | 管理者 | ステータス更新 |

| `PATCH` | `/facilities/admin/:id` | 管理者 | 施設更新 |

### 3.2 `POST /reservations` - 予約登録

**リクエストボディ**:
```typescript
{
  memberName: string;      // 必須, 1〜50文字
  email: string;           // 必須, RFC5322形式
  facilityId: string;      // 必須, 既存facilityIdのいずれか
  date: string;            // 必須, "YYYY-MM-DD", 当日以降
  startTime: string;       // 必須, "HH:MM", 施設営業時間内
  participants: number;    // 必須, 1〜施設定員
  purpose: string;         // 必須, 1〜200文字
  remarks?: string;        // 任意, 最大500文字
}
```

**処理フロー（トランザクション内）**:
1. バリデーション（形式・必須チェック）
2. 施設マスタ取得・営業時間チェック
3. `endTime` を `startTime + slotDurationMinutes` で自動計算
4. 重複予約チェック（同一施設・同一日・時間帯の重複 AND 定員超過）
5. Firestoreに`status: 'pending'`で保存（トランザクション）
6. AuditLog書き込み
7. 非同期でメール通知（失敗してもレスポンスは成功で返す）

**レスポンス（成功 201）**:
```typescript
{
  success: true;
  reservationId: string;
  message: "予約を受け付けました。確認メールをお送りしました。";
}
```

**エラーケース**:
| HTTPステータス | エラーcode | 意味 |
|---|---|---|
| 400 | `VALIDATION_ERROR` | バリデーション失敗（詳細をfieldsに格納） |
| 400 | `FACILITY_NOT_FOUND` | 存在しない施設ID |
| 400 | `OUTSIDE_OPERATING_HOURS` | 営業時間外 |
| 409 | `DUPLICATE_RESERVATION` | 重複予約 |
| 409 | `CAPACITY_EXCEEDED` | 定員超過 |
| 500 | `INTERNAL_ERROR` | 予期せぬエラー |

```typescript
// エラーレスポンス共通形式
{
  success: false;
  error: {
    code: string;
    message: string;          // ユーザー向けメッセージ（日本語）
    fields?: Record<string, string>; // バリデーションエラー時
  }
}
```

## 4. セキュリティ設計

### 4.1 入力バリデーション

- 全APIは Functions 内でバリデーション実施（クライアント側のみの検証に依存しない）
- `zod` ライブラリでスキーマ定義し、パースエラーを自動で400に変換
- メールアドレスは正規表現チェック後、内部では保存のみ（マスキング不要）
- テキスト項目はHTMLエスケープ不要（NoSQL + Reactは自動エスケープ済み）

### 4.2 管理者認証

- Firebase Authentication のIDトークンを `Authorization: Bearer {token}` で送信
- Functions内で `admin.auth().verifyIdToken()` を実行
- トークン検証後、Firestoreの`/admins/{uid}`ドキュメント存在確認で管理者権限確認

### 4.3 CORS設定

- Functions に `cors` ミドルウェアを適用
- 本番では `allowedOrigins` を Firebase Hosting のドメインのみに制限

### 4.4 レートリミット

- Cloud Armor（無料枠）または Functions の自前カウンターで1IPあたり10req/minを目安に設定
- 予約フォームはhoneypotフィールドでbot対策（カジュアルなボット除去）

## 6. 費用見通し

### 6.1 Blaze 前提での見方

Cloud Functions v2 の本番デプロイでは `cloudbuild.googleapis.com` などの有効化が必要になるため、本システムは実運用上 Blaze プラン前提で考える。

そのうえで、月額が高額になる構成ではなく、小規模運用なら無料枠相当かごく小額で収まる可能性が高い。

### 6.2 小規模運用時の概算

| リソース | 無料枠 | 月間想定使用量 | 判定 |
|---|---|---|---|
| Hosting 転送量 | 10GB/月 | 〜300MB | ✅ |
| Functions 呼び出し | 2M回/月 | 〜5,000回 | ✅ |
| Functions CPU時間 | 400,000秒/月 | 〜100秒 | ✅ |
| Firestore 読み取り | 50,000回/日 | 〜500回 | ✅ |
| Firestore 書き込み | 20,000回/日 | 〜50回 | ✅ |
| Firestore ストレージ | 1GB | 〜数MB | ✅ |

→ 月間予約500件・管理操作1,000回程度なら、Blaze を有効にしていても実費は低額に収まる見込み

### 6.3 課金リスクの対策

- Firebase コンソールで予算アラートを設定（上限: ¥500/月）
- Functions の `maxInstances` を設定（デフォルト無制限をDDoS対策で10に制限）
- Resendメール: 無料枠3,000通/月を超える場合のみ有料プラン検討

## 7. CI/CDパイプライン

```yaml
# .github/workflows/deploy.yml（概略）
on:
  push:
    branches: [main]

jobs:
  test:
    - npm ci
    - npm run lint
    - npm run test

  deploy-functions:
    needs: test
    - firebase deploy --only functions

  deploy-hosting:
    needs: test
    - npm run build
    - firebase deploy --only hosting
```

- PRマージ前にテストが必須
- `develop` ブランチへのpushはエミュレータテストのみ実行
- `main` への直接pushは禁止（ブランチ保護ルール）
